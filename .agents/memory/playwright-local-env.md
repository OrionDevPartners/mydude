name: Local Playwright in this container
description: Why the local Chromium browser backend is unavailable in the Replit container and what the real path is.
---

Installing the Playwright Chromium build succeeds (`python -m playwright install chromium` downloads to the workspace `.cache/ms-playwright`, which is gitignored), but the binary fails to launch with `error while loading shared libraries: libnspr4.so: cannot open shared object file`. The headless-shell needs system libs (nspr/nss/etc.) not present by default.

**How to apply:** Treat the local Playwright backend as optional/best-effort. Browserbase (cloud Chromium over CDP) is the production browsing path — it uses the same Playwright `page.route`/`goto` API, so logic verified against the contract holds there. Don't block delivery on a live local-browser test; verify policy/route logic with unit/route suites instead. If a local browser is truly needed, install the missing system libraries via the package-management skill (system deps), not just the playwright npm/pip install.

**Redirect block live-validated (2026-06-11):** an allowed→disallowed redirect (httpbin.org `redirect-to` → example.com, with only httpbin in the allow-list) was driven over real Browserbase/CDP and came back `blocked=True` with empty text. Non-obvious finding: httpbin's 302 resolved WITHOUT an intercepted `is_navigation_request()` hop, so it was the **post-navigation final-host check** in `_extract_page` (not the `page.route` interceptor) that caught it — both layers are load-bearing; don't drop the post-nav check as "redundant". Offline coverage lives in `tests/test_browser_redirect_failover.py` (fake-page interceptor abort, missed-hop post-nav, no-failover-on-block for both open_page and the interactive loop); the live test in that file auto-skips without Browserbase creds.

**Live-validated (2026-06-07):** the Browserbase path was driven end-to-end against a real two-step (email→password) SPA sign-in — login + account-view succeeded, policy allow-list gating held. The two-phase *cancel* was NOT live-tested (it is irreversible; no throwaway cancellable account). Honest "needs you" reporting (CAPTCHA/OTP/SSO) is correct in the contract suite. **Non-obvious gotcha:** real sign-in pages are SPAs whose fields are absent at `domcontentloaded` and hydrate a beat later — the login heuristic must `wait_for_selector` the form to mount before filling, or it silently no-ops and misreports "no password field / needs you" (fixed in `src/browser/backends.py` `_do_login`/`_wait_for_login_form`).

**Cancel heuristics — still unvalidated against real pages.** The cancel-button heuristics (`DEFAULT_CANCEL_TEXTS` + `_do_cancel` in `src/browser/backends.py`) and the two-phase gate (`request_cancel`/`confirm_cancel` in `src/subscriptions/manager.py`) are only verified offline (fake page + fake DB/broker; see `tests/test_subscription_cancel.py`). **Why still offline:** validating live needs throwaway *paid* accounts on Spotify/Netflix-style providers (payment method, email/CAPTCHA verification to create) and a genuinely cancellable subscription, plus running an irreversible cancel against a real account — none of which an agent can provision autonomously. Account-view login *is* live-validated; cancel is not. The account/billing page is also an SPA, so `_do_cancel` now waits for `networkidle` + the first cancel control to mount before scanning, and the fallback locator includes `[role=button]`/`<a>` (many providers style the cancel control as a link, not a real `<button>`). Real labels still need tuning once observed live.
